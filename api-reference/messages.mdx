## Endpoint

```
POST https://llm.orchid.ac/v1/messages
```

Anthropic-compatible **Messages** API. The same upstream model path is used as for [`/v1/chat/completions`](/api-reference/chat-completions); only the request/response wire format differs.

**Format:** Errors use the same **OpenAI-compatible** `error` object as chat completions (`message`, `type`, `code`). The Anthropic SDK may not map these into native Anthropic error types — inspect **raw HTTP** responses or logs when debugging failures.

## Request

### Headers

| Header | Value |
|--------|--------|
| `x-api-key` | Your Orchid key (`orchid-…`) — typical for the Anthropic SDK |
| `anthropic-version` | **`2023-06-01`** (required) |
| `Content-Type` | `application/json` |

`Authorization: Bearer <orchid-…>` is also accepted with the same key.

### Optional Orchid headers

There is **no** `orchid` object in the JSON body on this route. Use headers instead:

| Header | Values |
|--------|--------|
| `x-orchid-dehallucinate` | `true` or `false` — set `true` to run the post-reply grounding judge on **non-streaming** completions |
| `x-orchid-regenerate` | `true` or `false` — set `true` to auto-correct unsupported claims where possible. Implies `x-orchid-dehallucinate: true`. |
| `x-orchid-thinking` | `true` or `false` |
| `x-orchid-thinking-effort` | `none`, `low`, `medium`, or `high` |

### Body

Use the standard Anthropic **Messages** request shape: `model`, `max_tokens`, `messages`, etc. The **`model`** field must be **`orchid01`** (see [GET /v1/models](/api-reference/models)).

## Response

Standard Anthropic Messages response, plus the same top-level **`orchid`** key used on [chat completions](/api-reference/chat-completions#response) and [responses](/api-reference/responses#response):

```json
{
  "id":      "msg_abc123",
  "type":    "message",
  "role":    "assistant",
  "model":   "orchid01",
  "content": [{ "type": "text", "text": "According to the filing..." }],
  "stop_reason": "end_turn",
  "usage": {
    "input_tokens":  1240,
    "output_tokens": 380
  },
  "orchid": {
    "mcp_server":              "orchid-finance",
    "mcp_iterations":          1,
    "mcp_tool_calls":          [],
    "dehallucinate_requested": true,
    "grounded":                true,
    "score":                   0.97,
    "flagged_spans":           [],
    "checked":                 true
  }
}
```

`mcp_server`, `mcp_iterations`, and `mcp_tool_calls` are always present — the built-in `orchid-finance` tool suite runs automatically here too, even though user-defined tool-use blocks aren't supported (see [Tool calling](/guides/tool-calling#mcp-audit-fields)). The `dehallucinate_requested`/`grounded`/`score`/`flagged_spans`/`checked`/`span_fixes` fields follow [Hallucination detection](/guides/hallucination-detection).

## Limitations

- **Tool-use** and **image** content blocks you define yourself are **not** supported on this façade yet. Use [`/v1/chat/completions`](/api-reference/chat-completions) or [`/v1/responses`](/api-reference/responses) for OpenAI-style tools and multimodal workflows. The built-in `orchid-finance` tools still run automatically on this route (see [Tool calling](/guides/tool-calling)).
- If a built-in tool call happens mid-response, the reply is returned non-streamed even if you requested streaming — same behaviour as the other two endpoints.
- For rate limits and `429` behaviour, see [Rate limits](/api-reference/rate-limits).
